# Prompt 教程 00：Prompt 是什么，怎么写才稳定

这篇教程的目标：让你能写出“稳定、可复用、可评估”的提示词（Prompt），而不是靠灵感随便写两句。

## 1. Prompt 的本质
Prompt 不是“随便问一句话”，而是你给模型的一份**任务说明书**。说明书越清楚、越可验证，结果就越稳定。

把模型想成一个“很强但很容易误解你意图的实习生”会更贴切：你需要把目标、约束、输入输出格式说清楚。

## 2. 一个好 Prompt 的 6 个组成部分
按重要性排序通常是：

1) **目标（Objective）**
你要它完成什么？例如“总结这份文档为 5 条要点”。

2) **输出格式（Output Format）**
你希望它怎么输出？例如“必须输出 Markdown 列表，且每条不超过 20 字”。

3) **上下文（Context）**
任务所需背景知识、规则、数据、文档片段等。

4) **约束（Constraints）**
不能做什么、必须遵守什么。例：“不能编造；不确定必须写‘未知’并说明缺什么信息”。

5) **角色（Role）**
把模型拉到一个合理的专业角色上：如“你是资深知识库维护者”。

6) **示例（Examples / Few-shot）**
给 1~3 个输入输出对，让模型模仿格式与风格。

> 实务建议：先把 1) 目标 与 2) 格式写清楚，再补上下文与约束；角色与示例是增强项。

## 3. 最常见的 4 类失败原因
- **目标不清**：你觉得“写个总结”很清楚，但模型会理解成“扩写、联想、补充背景”。
- **格式不明确**：你希望 JSON，它给你段落；你希望 5 条，它写了 12 条。
- **上下文不足**：缺少关键输入时，模型会倾向“补全空白”（也就是幻觉风险）。
- **约束缺失**：没说“不得编造”，它就会“尽力回答”。

## 4. 你可以直接复用的 Prompt 骨架
你后续写 Prompt 可以统一用这个骨架（建议你存到 `prompts/templates/` 里）：

- 任务：……
- 输入：……
- 输出格式：……
- 约束：……
- 评估标准（可选）：……

## 5. 与 RAG/Tool 的关系
- Prompt：定义任务与格式（让模型“按规则说话”）
- RAG：补充可追溯的外部资料（让模型“有依据”）
- Tool：提供可执行能力（让模型“真的能做事”）

当你把这三者组合起来时，Prompt 负责“调度与验收标准”。
